# 别再让 AI 自动总结 Skill 了！

> 知乎专栏文章，讲透 Skill 的正确设计方法：把重复流程拆成 SKILL.md、references/ 和 scripts/，而非让 AI 一次对话总结成一个长 Prompt。

:```ascii

                      ┌─ Skill 标准结构的职责边界 ──────────────────────┐
                      │                                                   │
                      │   my-skill/                                       │
                      │   ├── SKILL.md        入口/流程/边界              │
                      │   ├── references/     配置/模板/示例/术语          │
                      │   └── scripts/        稳定/机械/确定性动作          │
                      │                                                   │
                      │   **核心原则**：                                   │
                      │   · SKILL.md 只写触发条件和执行路线                │
                      │   · references/ 放需要参考但不适合塞入入口的材料    │
                      │   · scripts/ 做解析/排序/去重/校验等稳定动作        │
                      │   · AI 只做判断/归纳/表达等需要语义理解的事        │
                      └───────────────────────────────────────────────────┘
:```

## 核心洞见

| 观点 | 说明 |
|------|------|
| ❌ **别让 AI 自动总结 Skill** | 一次成功对话混着流程/偏好/模板/临时上下文，AI 总结出来的是长篇 Prompt 而非可复用 Skill |
| ✅ **Skill = 工作目录** | `SKILL.md` 做入口，`references/` 存材料，`scripts/` 跑稳定动作 |
| 🎯 **description 不是宣传语** | "这是一个强大的多功能 Skill" 触发信号弱；应写"当用户需要 XXX 时使用" |
| 🤖 **AI 只做判断类工作** | 值不值得读、归哪类、怎么写清楚 → AI；解析/排序/去重 → 脚本 |
| 📋 **自查清单** | 写完 Skill 后检查 10 个问题，一半答不上来就还是长 Prompt |

## 适用场景

| 场景 | 说明 |
|------|------|
| 🆕 初次写 Skill | 避免掉入"让 AI 自动总结"的常见陷阱 |
| 🔧 Skill 重构 | 把已有的长 Prompt 式 Skill 拆成标准结构 |
| 🏗️ Skill 设计评审 | 用文中的自查清单检查 Skill 质量 |
| 📚 团队 Skill 规范 | 统一团队内部的 Skill 结构和责任划分标准 |

## 常见误区

1. **"只写一个 SKILL.md 行不行"** — 简单任务可以，出现大量模板/配置就该拆
2. **"什么动作必须写脚本"** — 稳定、机械、可验证的强烈建议脚本化
3. **"AI 自动总结完全不能用吗"** — 可作为草稿，但不要当成最终 Skill
